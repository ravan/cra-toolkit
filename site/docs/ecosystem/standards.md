# Standards & Formats

The CRA toolkit consumes and produces artifacts in industry-standard formats. This page describes each standard, its relevance to CRA compliance, and how the toolkit uses it.

---

## OpenVEX

**Vulnerability Exploitability eXchange** — a minimal, JSON-based format for communicating whether a vulnerability is exploitable in a specific product context.

OpenVEX is the default output format of [`cra vex`](../tools/vex.md). Each OpenVEX document contains a context (document metadata including author, role, and timestamp) and one or more statements. Each statement associates a vulnerability (identified by CVE) with one or more products (identified by PURL) and assigns a status: `not_affected`, `affected`, `fixed`, or `under_investigation`. When the status is `not_affected`, a justification is required (such as `component_not_present`, `vulnerable_code_not_in_execute_path`, or `inline_mitigations_already_exist`).

**Minimum elements:** context, author, timestamp, statements with vulnerability + products + status.

**CRA relevance:** OpenVEX enables manufacturers to communicate exploitability assessments as required by Annex I, Part II — documenting which vulnerabilities are applicable to the specific product configuration being shipped. Downstream consumers can automatically filter vulnerability noise using VEX statements, focusing remediation effort on genuinely exploitable issues.

**Toolkit usage:** Default output of `cra vex`. Consumed by `cra policykit` for VEX coverage checks, by `cra csaf` for advisory enrichment, and by `cra evidence` for inclusion in the compliance bundle.

## CSAF 2.0

**Common Security Advisory Framework** — an OASIS standard for machine-readable security advisories. CSAF 2.0 defines a JSON schema for structured advisory documents containing vulnerability details, affected products, severity scores, remediation guidance, and distribution metadata.

A CSAF advisory contains: document metadata (publisher, tracking, distribution), a product tree (hierarchical component identification with PURLs), vulnerability entries (CVE data, scores, remediations, threats), and notes. The CSAF VEX profile extends the base schema with exploitability status fields.

**CRA relevance:** Article 14(8) requires manufacturers to inform users about exploited vulnerabilities in a structured, machine-readable format. CSAF 2.0 is the primary format for meeting this obligation — it is designed specifically for automated advisory processing and aggregation. Annex I, Part II, point (8) further requires security updates to be accompanied by advisory messages.

**Toolkit usage:** Output of `cra csaf`. Alternate output format of `cra vex` (via `--output-format csaf`). Consumed by `cra evidence` for inclusion in the compliance bundle. Accepted as VEX input by `cra vex` (upstream VEX) and `cra csaf`.

## CycloneDX

**OASIS SBOM standard** — a lightweight, security-focused software bill of materials format. CycloneDX represents software components, their relationships, licenses, and vulnerabilities in a JSON or XML schema.

CycloneDX JSON documents are identified by the `bomFormat: "CycloneDX"` field. The toolkit supports CycloneDX across all tools that accept SBOM input. Component identification relies on the `purl` (Package URL) field for precise cross-referencing between the SBOM and vulnerability data.

**CRA relevance:** Article 14 and Annex I require manufacturers to identify and document the components in their products. CycloneDX SBOMs provide the component inventory that underpins vulnerability assessment, advisory generation, and compliance evaluation.

**Toolkit usage:** Accepted as SBOM input by `cra vex`, `cra csaf`, `cra report`, `cra policykit`, and `cra evidence`. Generated by ecosystem tools like syft and cdxgen.

---

## SPDX

**Linux Foundation / ISO SBOM standard** (ISO/IEC 5962:2021) — a comprehensive software bill of materials format with strong provenance for license compliance and supply chain transparency.

SPDX JSON documents are identified by the `spdxVersion` field (e.g., `SPDX-2.3`). SPDX represents packages, files, relationships, and annotations. The toolkit maps SPDX packages to PURL identifiers for vulnerability correlation.

**CRA relevance:** As an ISO standard, SPDX carries additional weight in regulatory contexts. It satisfies the same CRA component identification requirements as CycloneDX and may be preferred by organizations with existing SPDX tooling or ISO compliance workflows.

**Toolkit usage:** Accepted as SBOM input by all toolkit tools alongside CycloneDX. Auto-detected — no format flag needed.

## SLSA

**Supply-chain Levels for Software Artifacts** — a framework for supply chain integrity, defining levels of assurance for how software artifacts are built. SLSA provenance attestations capture the builder identity, source repository, build type, and configuration used to produce an artifact.

SLSA provenance documents follow the in-toto attestation format with a `https://slsa.dev/provenance` predicate type. They provide cryptographic evidence linking a built artifact back to its source and build process.

**CRA relevance:** CRA Annex I requires manufacturers to ensure the integrity of their products through secure development practices. SLSA provenance attestations provide verifiable evidence that builds are reproducible, tamper-resistant, and traceable to source — supporting conformity assessment under Annex VII.

**Toolkit usage:** Evaluated by `cra policykit` policy CRA-AI-3.1, which checks for the presence and completeness of SLSA provenance attestations. Included in evidence bundles via `cra evidence`.

## SARIF

**Static Analysis Results Interchange Format** — an OASIS standard for the output of static analysis tools. SARIF provides a uniform JSON schema for reporting findings from security scanners, linters, and code analyzers.

SARIF JSON documents are identified by the presence of a `runs` array at the top level, with each run containing tool information and results. The format is tool-agnostic — any analyzer that outputs SARIF can feed into the CRA toolkit.

**CRA relevance:** SARIF enables the CRA toolkit to consume vulnerability findings from a broad ecosystem of analysis tools beyond dedicated vulnerability scanners. This supports Annex I requirements for thorough vulnerability identification using diverse analysis methods.

**Toolkit usage:** Accepted as scan input by `cra vex` and `cra csaf`. Produced by tools including CodeQL, Semgrep, and various SAST/DAST tools. Auto-detected alongside Grype and Trivy JSON formats.

## OPA / Rego

**Open Policy Agent** — a general-purpose policy engine that evaluates policies written in Rego, a declarative query language designed for policy decisions over structured data.

Rego policies express compliance rules as data-driven queries. Each policy evaluates input documents (SBOMs, VEX, scan results, provenance attestations) and produces structured results with pass/fail/warn status, descriptions, and remediation guidance.

**CRA relevance:** OPA/Rego provides the evaluation engine for mapping CRA Annex I requirements to automated, repeatable compliance checks. The declarative nature of Rego policies means compliance rules are auditable, version-controlled, and extensible — organizations can add custom policies alongside the toolkit's 10 built-in CRA policies.

**Toolkit usage:** Powers the [`cra policykit`](../tools/policykit.md) evaluation engine. The toolkit embeds 10 default CRA compliance policies covering Annex I requirements. Custom policies can be loaded from a directory via the `--policy-dir` flag. Rego files are embedded in the binary at build time for zero-dependency deployment.
