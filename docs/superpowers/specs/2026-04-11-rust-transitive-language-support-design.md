# Rust Transitive Language Support ŌĆö Design

Status: draft
Date: 2026-04-11
Supersedes: ŌĆö

## 1. Overview

Add production-grade Rust support to the transitive cross-package reachability analyzer. The foundation has already been laid by the previous refactor: a `LanguageSupport` interface lives in `pkg/vex/reachability/transitive/language.go`, per-language implementations live under `transitive/languages/<lang>/`, and the analyzer selects one via `LanguageFor`. Adding Rust means one new language subpackage, one new fetcher (`crates.io`), one `LanguageFor` case, and one `buildFetchers` case ŌĆö plus a small number of targeted extensions to the shared foundation that the Python/JavaScript work did not need but Rust does.

The design goal is accuracy, not expedience. The Rust LLM judge score must match the Python and JavaScript bar (Ōēź 9), and the analyzer must produce verdicts on real crates.io dependency chains without resorting to heuristics that the Rust type system would reject.

## 2. Goals and non-goals

### 2.1 Goals

- Full `LanguageSupport` implementation for Rust, registered in `LanguageFor`.
- `CratesFetcher` implementation of the `Fetcher` interface backed by the crates.io JSON API, with on-disk cache parity to `PyPIFetcher` and `NPMFetcher`.
- Accurate export enumeration for Rust library crates: bare `pub` visibility only, module-tree walk rooted at `lib.rs`, canonical-path keys plus re-export (`pub use`) resolution to a fixed point.
- Cross-file trait-impl state correctly bridged into the shared `RunHop` pipeline, enabling `&dyn Trait` dispatch resolution across compilation units.
- Parity with Python/JavaScript on regression gates: `task test`, `task test:transitive`, `task test:reachability:transitive:llmjudge`.
- A new LLM judge test case exercising a real-world Rust vulnerability, scoring Ōēź 9 against the judge.

### 2.2 Non-goals

- Macro expansion. Tree-sitter does not expand macros. Code generated by `#[derive(...)]` (e.g. `serde_derive`) is analyzer-invisible. Documented as a known limitation; addressed in a future work item, not this spec.
- `cfg`/feature evaluation. Tree-sitter parses all branches regardless of feature gates. The analyzer treats the crate as if every feature is enabled ŌĆö the conservative direction for reachability.
- Workspace-level analysis. The fetcher retrieves individual `.crate` tarballs from crates.io; each is analyzed in isolation. Workspace members are treated as separate crates, which matches how crates.io distributes them.
- Binary crates as *vulnerable callees*. A crate with only `main.rs` and no `lib.rs` exposes no public API to link against, so the transitive reachability question does not apply. Handled as an explicit degradation (not a silent skip). Binary crates are still valid as *callers* (application source).
- Const/type-level reachability. Only function, method, struct, enum, and trait items participate in the call graph.

### 3.2 Capability interfaces added to the foundation

Two optional interfaces are added to `transitive/language.go`. Languages implement them when they need the capability; the shared pipeline type-asserts and delegates only when present. Python and JavaScript do not implement either, so their behavior is unchanged.

```go
// ExportLister is an optional capability for languages whose public API
// enumeration cannot be expressed by the generic IsExportedSymbol +
// ModulePath + SymbolKey triple. Rust implements this to walk the
// `pub mod` tree from lib.rs and resolve `pub use` re-exports.
type ExportLister interface {
    ListExports(sourceDir, packageName string) ([]string, error)
}

// CrossFileStateExtractor is an optional capability for language
// extractors that accumulate state across files (e.g. trait-impl maps
// for dynamic dispatch resolution). RunHop snapshots state after each
// per-file symbol extraction and restores the merged snapshot before
// call extraction, ensuring cross-file dispatch resolves correctly.
type CrossFileStateExtractor interface {
    SnapshotState() any
    RestoreState(any)
}
```

Type assertions in the shared code:

- `exports.go`:
  ```go
  if lister, ok := lang.(transitive.ExportLister); ok {
      return lister.ListExports(sourceDir, packageName)
  }
  // existing generic walker path
  ```
- `hop.go`: `RestoreState` is defined as idempotent and additive ŌĆö calling it with a snapshot merges that snapshot's state into the extractor's live state without overwriting. The Rust extractor's existing `RestoreTraitImpls` already uses `appendUnique` semantics, so surfacing it on the interface is a rename away from working. This framing avoids a language-agnostic `mergeStates` helper that would need to know every language's state shape.

  `RunHop` snapshots after each per-file `ExtractSymbols` (Phase 1) and replays the full list before the call-extraction loop (Phase 3):

  ```go
  // Phase 1: per-file symbol extraction, snapshotting state as we go.
  var snapshots []any
  stateful, hasCrossFileState := ext.(transitive.CrossFileStateExtractor)
  for _, pr := range parseResults {
      symbols, _ := ext.ExtractSymbols(pr.File, pr.Source, pr.Tree)
      // ... existing handling ...
      if hasCrossFileState {
          snapshots = append(snapshots, stateful.SnapshotState())
      }
  }
  // Replay all snapshots into the extractor before call extraction.
  if hasCrossFileState {
      for _, s := range snapshots {
          stateful.RestoreState(s)
      }
  }
  // Phase 3: call extraction, now with the full cross-file trait-impl map.
  ```

### 3.3 `treesitter.Symbol` extension

```go
type Symbol struct {
    // ... existing fields ...
    IsPublic bool // true when the extractor can prove external visibility
}
```

Rust's extractor sets `IsPublic = true` only when a `visibility_modifier` child exists whose literal text is exactly `pub` (not `pub(crate)`, `pub(super)`, `pub(in path)`). Trait-impl methods are a special case: `impl Trait for Type` method items cannot carry a visibility modifier of their own, so the extractor sets `IsPublic = true` on every method inside a trait-impl block unconditionally. The real gatekeeping for trait-impl methods happens in the `ListExports` reachability pass (section 4), which only emits a key for a trait-impl method when the implementing `Type` is itself public and reachable from `lib.rs`.

Python and JavaScript leave `IsPublic` at its zero value; their `IsExportedSymbol` implementations do not read it. Rust's `IsExportedSymbol` reads it, but is only consulted as a fallback ŌĆö the primary export enumeration path for Rust is `ListExports` (section 4), not the generic walker in `listExportedSymbols`. Implementing `IsExportedSymbol` nevertheless keeps the `LanguageSupport` contract honest and gives the generic walker a correct answer if a future code path ever reaches it for Rust.

## 4. Callee-side pipeline ŌĆö `rust.ListExports`

`ListExports` is the load-bearing accuracy component. Its input is the unpacked `.crate` source directory and the crate name; its output is the deduplicated set of dotted symbol keys by which downstream code can reach every exported item.

### 4.1 Stage 1 ŌĆö Module tree construction

Start at `<sourceDir>/<cratename>-<version>/src/lib.rs`. The crate tarball always unpacks into a `<name>-<version>/` root; the version suffix is known because `collectVulnSymbols` provides the pinned version. If `src/lib.rs` is absent, return `([], errNoLibraryAPI)`. The caller (`collectVulnSymbols`) translates this into `ReasonNoLibraryAPI`.

Parse `lib.rs`. The module tree walk produces a `[]moduleNode` where each node records:
- `Path []string` ŌĆö module path from the crate root (empty slice for `lib.rs` itself).
- `File string` ŌĆö absolute path to the module's source file.
- `IsPublic bool` ŌĆö true iff every ancestor in the chain was declared with `pub mod` (public-reachability property).

Algorithm:

1. Seed the queue with the root node `{Path: [], File: src/lib.rs, IsPublic: true}`.
2. Dequeue a node, parse its file, and scan for `mod_item` nodes at module scope.
3. For each `mod_item`:
   - If it carries a `visibility_modifier` whose text is exactly `pub`, and the parent module's `IsPublic` is true, the child's `IsPublic` is true.
   - Otherwise the child's `IsPublic` is false. Continue walking (we need the source for re-export analysis of re-exports *into* public modules from private ones), but symbols inside the private module are not emitted directly.
   - Inline modules (`pub mod foo { ... }`) are walked by recursing into the `declaration_list` child without needing filesystem resolution.
   - File-backed modules (`pub mod foo;`) are resolved by trying, in order:
     1. `<parent_dir>/<foo>.rs`
     2. `<parent_dir>/<foo>/mod.rs`
     3. `<parent_dir>/<foo>/<foo>.rs` (legacy 2018-edition-minus-one form)
     The first file that exists wins. If none exists, emit a debug log and skip ŌĆö this is a fetcher-produced tarball issue, not an analyzer issue.

4. Each discovered child is enqueued with the extended path.

The walk terminates when the queue is empty. The result is the set of `(Path, File, IsPublic)` tuples forming the canonical module tree.

### 4.2 Stage 2 ŌĆö Canonical export enumeration

For each publicly accessible module node (`IsPublic == true`), parse its file once and enumerate:

- `function_item` nodes at module scope whose visibility is exactly `pub`.
- `struct_item` and `enum_item` nodes whose visibility is exactly `pub`.
- `trait_item` nodes whose visibility is exactly `pub`.
- `impl_item` blocks on publicly reachable types: for inherent impls (`impl Type { ... }`), emit every `pub fn` method inside. For trait impls (`impl Trait for Type { ... }`), emit every method inside if both `Trait` and `Type` refer to types reachable from the public API (same crate `pub` type or a foreign public type).
- `const_item` and `static_item` are excluded ŌĆö they cannot appear as call targets.

For each emitted item `Item` in a module with canonical path `cratename.mod1.mod2`, the canonical key is `cratename.mod1.mod2.Item`. For inherent-impl methods the key is `cratename.mod1.mod2.Type.method`. For trait-impl methods the key is `cratename.mod1.mod2.Type.method` (the type, not the trait ŌĆö callers invoke methods on values of the type).

### 4.3 Stage 3 ŌĆö Re-export (`pub use`) resolution

Re-exports are the reason Rust cannot use the flat key scheme that worked for JavaScript and cannot use the purely hierarchical scheme that worked for Python. Real crates like `tokio` expose `tokio::spawn` as a re-export of `tokio::task::spawn`; downstream callers write `use tokio::spawn;` and never see the inner path. Both paths must match.

**Collection.** For each publicly accessible module at path `M`, scan for `use_declaration` nodes that carry a `visibility_modifier` with text exactly `pub`. Handle all four `use` forms that the existing Rust extractor already parses:

- Simple: `pub use source::path::Item;` ŌĆö one re-export `(M.Item ŌåÆ canonical of source::path::Item)`.
- Grouped: `pub use source::path::{A, B, C};` ŌĆö three re-exports sharing the same source prefix.
- Aliased: `pub use source::path::Item as Renamed;` ŌĆö re-export `(M.Renamed ŌåÆ canonical of source::path::Item)`.
- Wildcard: `pub use source::path::*;` ŌĆö for every canonical symbol whose canonical path begins with `source::path::`, emit `(M.<tail> ŌåÆ canonical)`.

Path prefixes in re-exports can be absolute (`crate::foo::bar`), relative (`self::foo::bar`, `super::foo::bar`), or foreign-rooted (`other_crate::foo::bar`). Only same-crate re-exports contribute to the key set ŌĆö foreign re-exports expose symbols of the foreign crate, which belong to that crate's key space, not ours. The walker skips any `pub use` whose root segment is a foreign crate name.

**Fixed-point resolution.** Re-exports can chain. `pub use a::Item;` followed by `pub use b::a;` where `b` lives in another module means the public path of `Item` includes both the direct form and every chain. The collection produces a directed re-export graph `E = {(public_path, canonical_path)}`; resolution iterates until no new edges are added, which is guaranteed to terminate because the canonical-path space is finite and deduplication is on exact string match.

**Output.** For each canonical symbol, the emitted key set is `{canonical_path} Ōł¬ {all re-export paths that transitively resolve to it}`. Deduplicated. Returned as the flat `[]string` that `listExportedSymbols` expects.

### 4.4 Why `ModulePath` and `SymbolKey` are still implemented

The `ExportLister` hook bypasses the generic `listExportedSymbols` walker, but `ModulePath` and `SymbolKey` are still invoked on the **caller side** by `RunHop` and the generic scope builder. The Rust implementation of both:

- `ModulePath(file, sourceDir, packageName)` ŌĆö returns the dotted module path from the crate root to the file, following the same `src/lib.rs` + file-resolution conventions used in Stage 1. For files outside the crate (tests/, examples/, benches/), it returns the first non-`src` path component so the generic prefix filter rejects them.
- `SymbolKey(modulePath, symbolName)` ŌĆö plain dotted concatenation: `modulePath + "." + symbolName`.

## 5. Caller-side pipeline

### 5.1 Imports ŌĆö `NormalizeImports`

The Rust extractor's `ResolveImports` already produces canonical `treesitter.Import` records for simple, grouped, aliased, and wildcard `use` declarations. However it preserves `::` in the `Module` field (e.g. `std::collections::HashMap`), while the symbol key scheme uses `.` throughout. `NormalizeImports` converts `::` ŌåÆ `.` in both `Module` and (defensively) `Alias` fields for every import. This is the only transformation applied.

### 5.2 Dotted target resolution ŌĆö `ResolveDottedTarget`

Identical semantics to Python/JavaScript. The extractor already normalizes `scoped_identifier` call targets from `Foo::bar::baz` to `Foo.bar.baz` before emitting edges. Resolution splits on the first `.`, looks up the prefix in scope, and substitutes the canonical module path on match. Fallback-to-original on miss.

### 5.3 Self-dispatch ŌĆö `ResolveSelfCall`

Rust method calls through `self` are emitted by the extractor as `self.method_name` when no better resolution is available. The rewrite logic is identical to Python's: if `to` has prefix `self.`, strip it, split `from` on `.`, take all but the last component as the class-qualifier, and emit `<qualifier>.<method_name>`. Because the Rust extractor sets `currentFunc = qualifyName(mod, typeName, methodName)` inside impl blocks, `from` will always have at least three components (`mod.Type.method`) when `self.X` is a valid rewrite target.

### 5.4 Cross-file trait dispatch

The existing Rust extractor already supports `&dyn Trait` dispatch within a single file: `ExtractSymbols` populates `traitImpls`, `methodToTypes`, and `paramTypes`; `processFieldCall` consults `traitImpls` to emit `EdgeDispatch` edges to every known implementor of a parameter's trait type. But `ExtractSymbols` **resets** this state at the start of every call, meaning a trait declared in `lib.rs` and implemented in `server.rs` loses its link before the caller in `client.rs` is processed.

The `CrossFileStateExtractor` hook (section 3.2) fixes this. The Rust extractor implements:

```go
func (e *Extractor) SnapshotState() any { return e.SnapshotTraitImpls() }
func (e *Extractor) RestoreState(s any) {
    snap, _ := s.(*TraitImplSnapshot)
    e.RestoreTraitImpls(snap)
}
```

`SnapshotTraitImpls` and `RestoreTraitImpls` already exist and already use `appendUnique` semantics; they just need to be surfaced on the interface. `RunHop` snapshots after each per-file `ExtractSymbols` and restores all snapshots before the Phase 3 `ExtractCalls` loop. Result: every file's trait-impl contribution is visible during every file's call extraction.

## 6. Crates.io fetcher

### 6.1 API contract

`CratesFetcher` implements `Fetcher` with ecosystem key `"crates.io"`. Backing endpoints:

- **Metadata.** `GET https://crates.io/api/v1/crates/{name}/{version}` returns JSON. The `version.dl_path` field is a relative path to the tarball download. The `version.checksum` field is the SHA-256 of the `.crate` file in lowercase hex.
- **Dependencies.** `GET https://crates.io/api/v1/crates/{name}/{version}/dependencies` returns `{"dependencies": [...]}` where each entry has `crate_id` (name), `req` (semver constraint), `kind` (`"normal"`, `"dev"`, or `"build"`), and `optional`. The fetcher maps only `kind == "normal"` entries to `PackageManifest.Dependencies`; dev and build deps are excluded because they do not appear in downstream compiled code. Optional dependencies are included (conservative choice ŌĆö we cannot tell which features the downstream consumer enabled).
- **Download.** The tarball is fetched from `https://crates.io{dl_path}`. The URL pattern is stable (`/api/v1/crates/{name}/{version}/download`) but we prefer the API-returned `dl_path` to avoid encoding assumptions.

### 6.2 `Fetch` implementation

Mirrors `PyPIFetcher.Fetch`:

1. Call `Manifest` to resolve metadata and compute the download URL and expected SHA-256.
2. Check the cache by digest string; short-circuit on hit.
3. Download the tarball; verify the SHA-256 matches the API-declared checksum and the caller's `expectedDigest` if non-nil.
4. Unpack using the shared `untarGz` helper (reused from `fetcher_pypi.go`). The `.crate` file is a gzipped tar whose single top-level directory is `<name>-<version>/`.
5. `SourceDir` points to the unpacked root (not the inner crate directory ŌĆö `ListExports` handles the `<name>-<version>/src/` resolution so that the fetcher stays ecosystem-agnostic).

Rationale for reusing `untarGz`: the existing helper lives on `fetcher_pypi.go` but performs no PyPI-specific logic. As part of this work it is promoted to a neutral file (`fetcher_zip.go` is already the companion for `.whl`; `untarGz` is moved to a new `fetcher_tar.go` or left where it is and called across files ŌĆö the choice is an implementation nit and will be decided during the code phase).

### 6.3 `Manifest` implementation

Two HTTP calls: metadata endpoint for checksum and download path, dependencies endpoint for the dependency map. Both are cacheable via HTTP `Cache-Control` semantics, but the in-process `Cache` is keyed on unpacked-artifact digest, not manifest JSON ŌĆö manifest calls remain uncached at this layer, matching PyPI and NPM.

### 6.4 Error mapping

- Network / non-200: wrap as `ReasonManifestFetchFailed` (metadata) or `ReasonTarballFetchFailed` (download).
- SHA-256 mismatch (API-declared vs downloaded): wrap as `ReasonDigestMismatch`.
- SHA-256 mismatch (caller-provided expected vs downloaded): wrap as `ReasonDigestMismatch` with the SBOM-expected marker.
- Tarball unpack failure: return a generic unpack error (not a named degradation; indicates a bug or a tampered artifact).
- `dl_path` missing in metadata: wrap as `ReasonManifestFetchFailed` with a sub-reason (the API contract guarantees this field; its absence means the crate is yanked in a way we do not recognize).

## 8. Testing strategy

### 8.1 Unit tests ŌĆö `languages/rust/`

`exports_test.go` covers every permutation of the export enumeration pipeline using hand-authored source trees written into `t.TempDir()`:

- Simple crate with one `pub fn` in `lib.rs`. Expects one canonical key.
- Crate with nested `pub mod foo; pub mod bar;` and one `pub fn` in each. Expects two keys with the full module paths.
- Crate with `mod private; pub mod public;`. `pub` items inside `private` do not leak; `pub` items inside `public` do.
- Crate with `pub mod foo { pub fn x() {} }` (inline module). Same semantics as file-backed.
- `pub mod foo;` resolved via `src/foo.rs` vs `src/foo/mod.rs` ŌĆö both resolution paths exercised.
- `pub(crate) fn` and `pub(super) fn` excluded from exports.
- `impl Type { pub fn m() }` emits `crate.Type.m`.
- `impl Type { fn m() }` does not emit (no `pub`).
- `impl Trait for Type { fn m() }` emits `crate.Type.m` (trait impl methods are effectively public if trait and type are public and reachable).
- `impl Trait for Type` where `Trait` is a foreign trait (e.g. `std::fmt::Display`) still emits the method (foreign trait ŌćÆ method is callable from external code that holds a `Type` value).
- `pub use inner::Foo;` at `lib.rs` level emits both `crate.inner.Foo` and `crate.Foo`.
- `pub use inner::{Foo, Bar};` (grouped) emits four keys: two canonical plus two re-exported.
- `pub use inner::Foo as Renamed;` emits `crate.inner.Foo` and `crate.Renamed`.
- `pub use inner::*;` emits the canonical plus one re-export per `pub` item in `inner`.
- Chained: `pub use a::Thing;` in `lib.rs`, `pub use b::Thing;` in module `a`, `pub struct Thing;` in module `b`. Expects all three paths: `crate.b.Thing`, `crate.a.Thing`, `crate.Thing`.
- Foreign `pub use serde::Serialize;` does not contribute a `crate.Serialize` key (it exposes serde's symbol, not ours).

`rust_test.go` covers:

- `Name`, `Ecosystem`, `FileExtensions`, `Grammar`, `Extractor` return expected values.
- `IsExportedSymbol` delegates to `IsPublic` for function/method/class symbols, returns false for other kinds.
- `ModulePath` handles `src/lib.rs`, `src/foo.rs`, `src/foo/mod.rs`, out-of-src files.
- `SymbolKey` is dotted concatenation.
- `NormalizeImports` converts `::` to `.` in `Module` and `Alias`.
- `ResolveDottedTarget` resolves via scope lookup with fallback.
- `ResolveSelfCall` rewrites `self.method` to `mod.Type.method` given a three-component `from`, leaves unchanged otherwise.

## 10. Edge cases and known limitations

- **Procedural and derive macros** are invisible to tree-sitter. Code generated by `#[derive(Serialize)]` is not in the tree. A vulnerability that lives inside generated code is not reachable from our perspective. Documented; future work.
- **`cfg` branches** are always parsed. A target symbol gated behind a feature the downstream consumer did not enable will still show as reachable if the static call path exists. Conservative; matches the "we report what the code *could* do" framing.
- **`impl Trait` return types** resolving to concrete types via type inference are not resolved ŌĆö we emit edges at the declared type level only. This mirrors the extractor's existing design.
- **`async fn`** desugars to `fn ... -> impl Future<Output = T>`. Tree-sitter sees the `async` keyword but does not restructure the call graph. Async calls appear as direct calls in our edges, which is correct for static reachability.
- **Workspace-internal path dependencies** (`foo = { path = "../foo" }`) are not published to crates.io, so the fetcher cannot resolve them. Affected deps produce `ReasonManifestFetchFailed`. Acceptable; the SBOM would not list unpublished path deps either.
- **Yanked versions** still return metadata from the API and can still be fetched; analysis proceeds normally. Yank status is not reported by this analyzer.
